**回答**: 每次调用时单独传入

## 技术方案选择

### 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案A: 单LLM** | 一次调用完成所有评估 | 简单、灵活 | 评分一致性难控制 |
| **方案B: 流水线** | JD解析→分项匹配→汇总，三步走 | 稳定、可调试 | 流程稍复杂 |
| **方案C: LLM+规则** | LLM提取要求 + 向量匹配 + LLM生成报告 | 平衡可控与灵活性 | 需要维护匹配规则 |

### 选择
**方案B (流水线)** - 理由：评估结果一致性更好，易于调试和优化各维度评分逻辑

## 设计方案确认

### 架构
- 三阶段流水线: JD解析 → 多维度匹配 → 报告生成
- 技术栈: Python 3.10+, FastAPI, Claude API

### 输出格式
- overall_score: 综合评分
- dimensions: 四个维度分项评分 (hard_skills, experience, education, soft_skills)
- recommendation: 推荐/不推荐
- reasons: 匹配理由列表

### 权重分配
- hard_skills: 40%
- experience: 30%
- education: 15%
- soft_skills: 15%
