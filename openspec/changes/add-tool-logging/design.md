## Context

HR Agent 当前使用 LangGraph 的 `create_react_agent` 和三个自定义工具(`parse_jd`, `score_candidate`, `generate_report_html`)。工具通过 LangChain 的 `@tool` 装饰器封装,在 ReAct 循环中被 LLM 顺序调用。

当前痛点:

- 无法观测工具调用顺序、耗时、状态
- 调试困难,性能瓶颈不明确
- 生产环境缺少监控能力

技术约束:

- 必须使用 Python 标准库,零新增依赖
- 不能影响现有 API 响应格式
- 性能开销需要最小化(<5ms)
- 需要同时支持开发和生产环境

## Goals / Non-Goals

**Goals:**

- 实现轻量级、结构化的工具调用日志系统
- 记录每个工具的开始/结束时间、执行时长、输入输出预览、异常
- 支持开发环境(人类友好)和生产环境(JSON)两种输出格式
- 通过装饰器实现,对工具代码侵入性最小

**Non-Goals:**

- 不实现分布式追踪(OpenTelemetry)
- 不实现日志聚合/分析平台
- 不记录 LLM 推理过程(仅工具调用)
- 不实现异步日志(避免增加复杂度)

## Decisions

### Decision 1: 使用装饰器模式而非 ToolNode.wrap_tool_call

**选择**: 装饰器 `@traced_tool` 直接应用于工具函数

**理由**:

- **简单性**: 装饰器模式在 Python 中更符合习惯,代码清晰
- **灵活性**: 可以独立于 LangGraph 使用,不依赖框架细节
- **可测试性**: 装饰器逻辑可以单独测试
- **未来扩展**: 后续可以轻松添加更多装饰器(如性能分析、缓存)

**备选方案及问题**:

- `ToolNode.wrap_tool_call`: 需要深入理解 LangGraph 内部机制,绑定过强
- LangChain callbacks: 需要实现 callback handler,更复杂,且只能捕获 LangChain 层事件

### Decision 2: 同步日志而非异步

**选择**: 日志操作同步执行(blocking)

**理由**:

- **简单性**: 无需引入异步队列、后台线程
- **可靠性**: 同步日志保证在异常发生时也能记录
- **性能可接受**: Python logging 模块的 StreamHandler 已经足够快(<1ms)
- **一致性**: 日志顺序严格对应执行顺序

**备选方案及问题**:

- 异步日志(queue + worker): 增加复杂度,可能丢失崩溃时的日志

### Decision 3: 日志格式通过 ENV 环境变量切换

**选择**: 根据 `ENV` 环境变量选择 `DeveloperFormatter` 或 `StructuredFormatter`

**理由**:

- **零配置**: 默认行为(开发模式)无需额外设置
- **明确性**: ENV=production 是广泛认可的约定
- **灵活性**: 可通过环境变量轻松切换,无需改代码

**备选方案及问题**:

- 配置文件: 增加文件管理负担
- 命令行参数: 不适合长期运行的服务

### Decision 4: 输入输出截断为 100 字符

**选择**: 自动截断超过 100 字符的输入输出,附加 "(truncated, N total chars)"

**理由**:

- **隐私保护**: 避免完整记录敏感信息(简历、JD)
- **日志体积**: 控制日志大小,防止爆炸增长
- **可读性**: 100 字符足够识别问题,不会淹没关键信息
- **调试需要**: 如需完整数据,可临时调整 max_len 参数

**备选方案及问题**:

- 完整记录: 隐私风险,日志体积巨大
- 脱敏(mask 姓名/电话): 增加复杂度,且很难穷举所有敏感字段

### Decision 5: 使用 functools.wraps 保持函数元数据

**选择**: 装饰器内部使用 `@wraps(func)` 保持原函数签名

**理由**:

- **兼容性**: LangChain 的 `@tool` 装饰器依赖函数 `__name__` 和 `__doc__`
- **调试友好**: 异常堆栈显示原函数名,不会显示 `wrapper`
- **工具链支持**: IDE、类型检查器可以正确识别函数

## Risks / Trade-offs

### Risk 1: 日志体积增长过快

**Mitigation**:

- 默认截断输入输出到 100 字符
- 生产环境使用日志轮转(可选,配置 RotatingFileHandler)
- 只记录工具调用,不记录 LLM 推理过程

### Risk 2: 同步日志可能轻微影响性能

**Mitigation**:

- 使用 StreamHandler(内存缓冲),避免直接磁盘 I/O
- 截断减少序列化开销
- 如测试发现>5ms 开销,考虑引入 BufferedHandler

### Risk 3: 敏感信息可能仍泄露到日志

**Mitigation**:

- 截断到 100 字符降低完整泄露风险
- 文档说明:生产环境日志应限制访问权限
- 后续可扩展:添加 PII 检测和自动脱敏

### Risk 4: 多工具并发调用时 call_id 可能冲突(虽然当前是顺序执行)

**Mitigation**:

- call_id 包含毫秒时间戳 + 工具名,冲突概率极低
- 如未来支持并发,可升级为 UUID

### Trade-off: 不实现异步日志

- **好处**: 简单、可靠、顺序一致
- **代价**: 轻微性能开销(预期<5ms,可接受)

### Trade-off: 不集成 OpenTelemetry

- **好处**: 零依赖、快速实现、轻量级
- **代价**: 未来需要分布式追踪时需要重构(但装饰器架构容易迁移)

## Migration Plan

**部署步骤**:

1. 创建 `app/utils/logger.py`,不影响现有代码
2. 为三个工具添加 `@traced_tool` 装饰器
3. 本地测试验证日志输出符合预期
4. 部署到生产环境,设置 `ENV=production`
5. 观察日志体积和性能影响

**回滚策略**:

- 移除 `@traced_tool` 装饰器即可完全回滚
- 无数据库变更,无需迁移

**验证方法**:

- 运行集成测试,检查日志输出包含 tool_start/tool_end 事件
- 验证日志不影响 API 响应内容和格式
- 测量工具执行时间增加<5ms

## Open Questions

1. **是否需要 session 级别的追踪?**
   - 当前方案聚焦工具级别
   - 如需追踪完整 session 生命周期,可后续扩展 hr_agent.py

2. **是否需要持久化日志(文件)?**
   - 当前默认输出到 stdout
   - 生产环境是否需要配置 FileHandler?依赖部署架构(容器日志收集 vs 本地文件)

3. **是否需要可配置的截断长度?**
   - 当前硬编码 100 字符
   - 是否需要通过环境变量调整?

4. **是否需要统计聚合功能?**
   - 当前只记录单次调用
   - 是否需要内存统计器(调用次数、平均耗时、错误率)?
